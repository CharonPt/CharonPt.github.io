从强网杯-core入门linux-kernel

发表于 2022-10-03 更新于 2023-03-27

前言

虽然这题不是那种最简单的入门题，但是对于已经比较熟悉用户态pwn的人来说这题的难度并不难，从这题能够学习到关于linux kernel pwn的很多相关知识

题目最先给的是一个.tar.gz文件，解包之后如下：

首先是关于附件的说明：

  1. .sh：启动脚本，里面会标明启动方法、保护措施等

  2. .cpio：打包文件，解包以后里面有文件系统，其中vmlinux是内核的二进制文件，.ko是存在漏洞的驱动文件，也就是所需要分析的二进制文件

    解包命令：cpio -idv < ./rootfs.cpio

    打包命令：find . | cpio -o --format=newc > ../new_rootfs.cpio

  3. bzlmage：镜像文件

  4. vmlinux：相当于libc，可以从里面找gadget

首先是start.sh

从这里可以看出开了kaslr保护

然后是.cpio文件，解包后里面会有.ko后缀的驱动，放到跟.sh文件相同目录下。还有init文件，该文件里面的命令在启动内核时会被执行，里面有一些比较重要的命令，比如装载驱动、设置用户权限等

如果解包.cpio后发现没有vmlinux文件的话，可以用linux内核源码中的script中的extract-vmlinux脚本从bzlmage中提取出vmlinux，命令：./extract-vmlinux ./bzImage > vmlinux（需要先从script中把脚本拿出来）

有vmlinux时就可以从里面提取gadget，ropper --file ./vmlinux --nocolor > gadget.txt（是我的问题吗ropper卡到嗨那样），得到gadget文件之后用grep去找就行（注意这里的--nocolor是必要的，否则得到的文件会包含用来显示颜色的字符，会影响搜索）

About 调试

gdb

设置gdb远程调试时的位数：set architecture i386:x86-64，可以写在~/.gdbinit里

gdb远程附加：target remote:1234

添加符号表：add-symbol-file ./core.ko base_addr，base_addr是驱动加载的基地址，可以在root权限启动内核时用lsmod命令查看到

以上这些内容其实也可以写到一个sh里，方便很多

poc编译

因为一般的内核镜像里是没有libc文件的，所以我们写好的c程序需要静态编译才能让内核去运行：gcc ./poc.c -o poc -static -w，如果里面用到的内联汇编是Intel的那还需要加个参数：gcc poc.c -o poc -static -masm=intel

编译好poc后将其打包进.cpio内即可

完整调试流程

  1. 修改init文件（或者/etc/init.d/rcS文件）的驱动启动权限，改成以root启动（方便找地址）
  2. 编译好poc，打包进.cpio文件里面
  3. 启动内核，lsmod查看驱动基址
  4. 另开一个终端起gdb，gdb ./vmlinux，然后设置架构set architecture i386:x86-64（这个可以写在~/.gdbinit里，就会默认设置），然后添加符号表add-symbol-file ./core.ko base_addr，然后远程连上去target remote :1234，连上去之后打断点，因为一般.ko都会留符号表，所以可以选择直接断函数名，然后c
  5. 在内核终端里运行poc，gdb就能断在断点处（要是没断下来就检查下是不是add-symbol-file时的基址错了，或者poc流程根本跑不到断点处等等）

关于内核题的思考

内核题跟普通题目的区别

最开始学pwn时接触的题目的目的一般是通过存在漏洞的程序拿到shell

而内核题的最终目的是提权，其本身就会给我们一个shell的环境，只是权限比较低，这时我们要做的是通过某种方法来让我们的权限提升到root，而这种方法便是通过驱动程序的漏洞。这时我们的攻击方法从连接远程服务器攻击其漏洞程序变成了在远程服务器上运行我们编写的与其驱动程序交互的程序，通过编写程序来跟驱动程序进行交互，然后利用其中存在的漏洞最终达到提权的目的。

所以在初学内核时这两者的区别一定要搞清楚，做内核题时我们可以编写任意程序发送到远程服务器上直接运行，因为我们本身就有一个低权限的shell。

修改权限

kernel调度一切系统资源，并为用户应用提供运行环境，那么相应的，应用程序的权限也是kernel管理的。在内核中会使用task_struct结构体来表示一个进程，里面就有关于权限的部分，用到了cred结构体，内核使用一个cred结构体来管理一个进程的权限，那么很自然地想到，如果能修改一个进程的cred结构体，那么就能改变一个进程的权限。

其中一个方法就是在内核态中执行commit_creds(prepare_kernel_cred(NULL))，在prepare_kernel_cred()函数中，若传入的参数为NULL，则会缺省使用init进程的cred作为模板进行拷贝，即可以直接获得一个标识着root权限的cred结构体，然后commit_creds()函数用来将一个cred结构体应用到进程。

那么内核题的流程就是：低权限程序 -> 与内核模块交互，进入内核空间让内核模块执行commit_creds(prepare_kernel_cred(NULL))，使当前进程变成root权限 -> 返回到用户空间，执行system('/bin/sh')新起一个shell（root权限）

跟内核模块进行交互

现在的问题就来到了怎么跟驱动程序进行交互，远程服务器会把驱动程序挂载到靶机上（对应的我们本地使用执行启动脚本时其也会将驱动程序挂载到qemu里）

Linux定义了系统调用ioctl供进程与设备之间进行通信，也就是说我们可以在程序中调用ioctl来跟某个驱动程序交互

系统调用ioctl是一个专用于设备输入输出操作的一个系统调用，其调用方式如下：

  • fd：设备的文件描述符
  • request：请求码
  • 其他参数

对于一个提供了ioctl通信方式的设备而言，我们可以通过其文件描述符、使用不同的请求码及其他请求参数通过ioctl系统调用完成不同的对设备的I/O操作

例如CD-ROM驱动程序弹出光驱的这一操作就对应着对“光驱设备”这一文件通过ioctl传递特定的请求码与请求参数完成

返回用户态的方法

使用swapgs;iretq指令返回用户态

swapgs指令简单地说就是将当前gs基址寄存器的值与一个特定地址（IA32_KERNEL_GS_BASE MSR）的值交换，从而快速获得指向内核空间/用户空间数据的指针，举个例子，程序运行在用户态时gs是用户空间的地址，此时IA32_KERNEL_GS_BASE MSR中保存的就是内核空间的地址，要进入内核态时执行swapgs，gs的值就变成了内核空间的地址，再需要返回到用户空间时同理。

iretq指令用来完成中断返回，其是中断服务程序中的最后一条指令，功能是将栈上的段地址、偏移地址pop到对应寄存器中，使程序返回原来发生中断的地方，其从栈顶弹出 EIP >> CS >> EFLAGS >> ESP >> SS （类似于RETF）

为什么需要手动返回用户态

正常情况下在程序中用利用ioctl函数去跟驱动进行交互时，函数结束完后当然会自动返回到用户态继续执行我们的程序，但是我们如果在内核态中利用栈溢出等漏洞劫持了PC指针之后，因为内核已经不走正常返回用户态的指令了，所以除了让其执行提权操作以外，我们还需要手动返回到用户态去开shell。

题目

了解一点kernel基础知识，再知道了前面说的东西之后我觉得就可以开始做题目了

这道题中存在这些函数

其中init_module()跟exit_core()是模块的入口/出口函数，在该模块被装载/卸载时会分别被调用

该函数注册了/proc/core，所以我们在poc中可以用open打开/proc/core，然后用其文件描述符对其进行交互。

相对的，exit_core()就是删除/proc/core

然后看到其定义的core_ioctl()函数，我们在程序中对/proc/core调用ioctl函数便是调用这个函数

用传入的第二个参数来决定执行哪些命令，分别是调用core_read()、设置全局变量off、调用core_copy_func()

可以看到core_read()函数其实就是从&v5[off]将0x40个字节拷到用户空间（我们传入的指针），因为off是我们可以设置的且其没有检查，所以很明显的数组越界，可以利用这个函数将内核栈上的canary拷贝到用户空间，那么我们就得到了canary

然后是core_copy_func()函数，相对的，其将自定义长度的数据从内核态的全局变量name中拷贝到内核栈上，这里虽然对长度进行了检查，但是存在负数绕过，所以这里是个栈溢出，同时注意最后qmemcpy时size取的是a1的低两字节。

全局变量name的数据从core_write()函数来，可以注意到core_ioctl()函数中并没有调用到core_write()，但是可以注意到该函数的参数跟普通的write()其实是一样的，我们可以在程序中直接write(fd,,)来调用这个函数。同时查看其交叉引用可以发现在数据段存在core_write()函数的地址，还有一些别的能在程序中被直接调用的函数

猜测这里跟能直接调这个函数有关，当然这些只是些题外话，做题时并不需要关心

到这里了其实思路就很清晰了

  1. 用core_read()拿到内核态的canary，为栈溢出做准备
  2. 找到相关gadget和函数地址用来写ROP chain
  3. 用core_write()往内核里写ROP chain
  4. 用core_copy_func()触发栈溢出控制内核PC指针执行ROP chain
