前言

ret2_dl_resolve

大概调用过程：

触发了_dl_runtime_resolve之后会

  1. 先到.rel.plt节，通过plt中给的push <偏移>在.rel.plt节中找到偏移处对应函数的重定位表项
  2. 再通过该表项的r_info（一个偏移量），找到.dynsym节中对应偏移量函数的符号表项
  3. 又通过该符号表项的st_name（也是一个偏移量）在.dynstr节找到对应的字符串
  4. 最后就会通过这个字符串（函数名）来调用该函数。

以下示例中的程序为32位，64位的流程未必一样

  1. call foo -》 jmp foo@plt

  2. foo@plt的内容一定是 jmp [foo@got]，意为取foo@got表项中的内容来跳转

  3. 在函数未进行过延迟绑定的解析时，foo@got表项中的内容会被填上foo@plt+6的地址，如下，其中0x804b22c是read@got的位置，此时里面的内容正是read@plt+6

    在IDA中观察一下plt表，可以发现所有函数的plt表项都是类似的操作：

    • X@plt+0处是jmp [X@got]
    • X@plt+6处是push index（这个index是该函数在rel.plt上的偏移.reloc_arg，_dl_runtime_resolve的第二个参数）
    • X@plt+B处是jmp plt0（plt0指的是plt表的第一项，是_dl_rumtime_resolve的表项）

  4. plt0中的内容是固定的

    • push [GOT + 4]（link_map结构的地址，_dl_runtime_resolve的第一个参数）
      • link map这个结构我们不关心，但是要知道它包含了.dynamic的指针，通过这个link_map，_dl_runtime_resolve函数可以访问到.dynamic这个section
    • jmp [GOT + 8]（_dl_runtime_resolve的地址），此时相当于开始执行_dl_runtime_resolve(link_map,reloc_arg)）

  5. 由_dl_runtime_resolve完成动态解析后将函数地址写回到foo@got，然后_dl_runtime_resolve在返回的时候就会调用这个函数（32位时是通过临近返回时push foo@addr，然后ret到这个地址来调用的）

到此为止省略了_dl_rumtime_resolve具体流程的延迟绑定流程结束

总结

最后再梳理一下整个解析流程（32位下）（跟源码中的步骤略有不同，但不重要）

  1. 基于link_map->l_info[]取得symtab和stratb的地址（这两个都是程序中的地址）
  2. 基于link_map->l_info[] + reloc_offset取得reloc（.rel.plt中的表项）
  3. 基于symtab[reloc->r_info >> 8]取得sym（symtab中的表项）
    • 这个地方注意会有一个reloc->r_info的assert，必须要让reloc->r_info的低1字节为0x7
  4. 基于strtab + sym->st_name取得对应函数字符串
  5. 基于函数字符串找到对应函数地址

  • 漏洞利用流程

    注意，在不伪造link_map的情况下，symtab，strtab，.rel.plt都是程序中定死的地址，所以我们只需要控制每一个步骤中对应的那块偏移值，就一定能让下一个目标结构落在我们想要的地方

  1. 控制 EIP 为 PLT[0] 的地址，只需要传递一个 reloc_arg 参数
  2. 控制 reloc_arg 的大小，使 reloc 的位置落在可控地址内
  3. 伪造 reloc 的内容，使 sym 落在可控地址内
  4. 伪造 sym 的内容，使 name 落在可控地址内
  5. 伪造 name 为任意库函数名称，如 system

因为 Sym 结构体是根据 &symtab[ELFW(R_SYM)(reloc->r_info)] 这样去找的，也就相当于 symtab + (reloc->r_info >> 8)*0x10 ，所以每个 Sym 结构体跟 .dynsym 节的起始地址的偏移都是按 0x10 个字节对齐的，即 (fake_sym_addr - symtab)%0x10 == 0 ，所以只要让 (fake_sym_addr - symtab)&0xf + x = 0x10 即可，这个 x 就是exp中作padding的align。

64位下正常伪造会crash的地方

64位不开PIE的程序中，程序段一般位于0x40000，数据段一般位于0x60000，而我们伪造的结构一般会塞在bss段，导致我们伪造的结构跟对应节区表的起始位置之间至少差掉了0x20000左右，这就导致我们伪造的reloc->r_info不可避免的会比较大。

源码中的这行通过*(vernum + sizeof(vernum) * (reloc->r_info >> 32)) & 0x7ffff来取ndx，但是这一部分已经碰到了一个未映射的段。由此正常情况下构造的话会碰到下面这个地方造成crash。

总结

综上，我们可以总结出两种思路

方法一（需要知道libc版本）

如果我们要绕过第一个if，即 if (__builtin_expect (ELFW(ST_VISIBILITY) (sym->st_other), 0) == 0)

那么我们最后会走的路径就是value = l->l_addr + sym->st_value，那么如何让value变成我们想要执行的函数呢？

我们需要

  1. 让l->l_addr设置成目标函数跟一个程序中已解析函数真实地址的偏移
  2. 让&(sym->st_value)为一个程序中已解析函数的got表的地址

但是，显然这种方法必须要知道libc版本，因为我们需要使用一个提前已知的函数偏移

现在假设我们期望执行system函数

例如程序中存在setbuf函数，且能在我们的攻击流程前完成解析，那么我们就需要让l->l_addr = system - setbuf

然后让sym + st_value = setbuf@got，即sym = setbuf@got - 8。这里相当于我们把sym放在了got表的附近，利用里面天然的数据来完成利用，而且正常情况下任何一个函数的got表项-8的位置都不可能是0，因为got0处会存link_map和dl_runtime_resolve的地址，也就天然绕过了第一个if。

基于这样的设置，value = offset + setbuf_addr = system_addr，我们就完成了利用

可以想到这样的方法相对来说方便一点，我们甚至不需要伪造sym结构，只要计算好reloc_arg即可。

但是也能想到既然我们需要伪造l->l_addr，那么我们逃不开要伪造link_map中必要的字段例如l->l_info[]，好在整个流程中用到的link_map的字段并不算多，我们只要伪造很小的一部分即可

方法二

方法二自然是正常进第一个if然后伪造link_map绕过第二个if，然后再接着像32位那样算好sym->st_name的值，并且布置字符串。

总的来说笔者认为工作量比方法一还要稍微大一些，毕竟方法一不需要往下伪造，而且二者都需要伪造link_map。

但是这个方法最关键的优点是它不需要知道libc版本，所以视具体情况，有时候还是不得选择这个方法
