  2. .cpio：打包文件，解包以后里面有文件系统，其中vmlinux是内核的二进制文件，.ko是存在漏洞的驱动文件，也就是所需要分析的二进制文件

  4. vmlinux：相当于libc，可以从里面找gadget

首先是start.sh

从这里可以看出开了kaslr保护

然后是.cpio文件，解包后里面会有.ko后缀的驱动，放到跟.sh文件相同目录下。还有init文件，该文件里面的命令在启动内核时会被执行，里面有一些比较重要的命令，比如装载驱动、设置用户权限等

如果解包.cpio后发现没有vmlinux文件的话，可以用linux内核源码中的script中的extract-vmlinux脚本从bzlmage中提取出vmlinux，命令：./extract-vmlinux ./bzImage > vmlinux（需要先从script中把脚本拿出来）

有vmlinux时就可以从里面提取gadget，建议用ropper，会快很多，ropper --file ./vmlinux > gadget，得到gadget文件之后用grep去找就行，或者直接用ropper加载（缺点是关掉终端就没了）

About 调试

gdb

设置gdb远程调试时的位数：set architecture i386:x86-64，可以写在~/.gdbinit里

gdb远程附加：target remote:1234

添加符号表：add-symbol-file ./core.ko base_addr，base_addr是驱动加载的基地址，可以在root权限启动内核时用lsmod命令查看到

poc编译

因为一般的内核镜像里是没有libc文件的，所以我们写好的c程序需要静态编译才能让内核去运行：gcc ./poc.c -o poc -static -w，如果里面用到的内联汇编是Intel的那还需要加个参数：gcc poc.c -o poc -static -masm=intel

编译好poc后将其打包进.cpio内即可

完整调试流程

  1. 修改init文件（或者/etc/init.d/rcS文件）的驱动启动权限，改成以root启动（方便找地址）
  2. 编译好poc，打包进.cpio文件里面
  3. 启动内核，lsmod查看驱动基址
  4. 另开一个终端起gdb，gdb ./vmlinux，然后设置架构set architecture i386:x86-64（这个可以写在~/.gdbinit里，就会默认设置），然后添加符号表add-symbol-file ./core.ko base_addr，然后远程连上去target remote :1234，连上去之后打断点，因为一般.ko都会留符号表，所以可以选择直接断函数名，然后c
  5. 在内核终端里运行poc，gdb就能断在断点处（要是没断下来就检查下是不是add-symbol-file时的基址错了，或者poc流程根本跑不到断点处等等）

关于内核题的思考

内核题跟普通题目的区别

最开始学pwn时接触的题目的目的一般是通过存在漏洞的程序拿到shell

而内核题的最终目的是提权，其本身就会给我们一个shell的环境，只是权限比较低，这时我们要做的是通过某种方法来让我们的权限提升到root，而这种方法便是通过驱动程序的漏洞。这时我们的攻击方法从连接远程服务器攻击其漏洞程序变成了在远程服务器上运行我们编写的与其驱动程序交互的程序，通过编写程序来跟驱动程序进行交互，然后利用其中存在的漏洞最终达到提权的目的。

所以在初学内核时这两者的区别一定要搞清楚，做内核题时我们可以编写任意程序发送到远程服务器上直接运行，因为我们本身就有一个低权限的shell。

修改权限

kernel调度一切系统资源，并为用户应用提供运行环境，那么相应的，应用程序的权限也是kernel管理的。在内核中会使用task_struct结构体来表示一个进程，里面就有关于权限的部分，用到了cred结构体，内核使用一个cred结构体来管理一个进程的权限，那么很自然地想到，如果能修改一个进程的cred结构体，那么就能改变一个进程的权限。

其中一个方法就是在内核态中执行commit_creds(prepare_kernel_cred(NULL))，在prepare_kernel_cred()函数中，若传入的参数为NULL，则会缺省使用init进程的cred作为模板进行拷贝，即可以直接获得一个标识着root权限的cred结构体，然后commit_creds()函数用来将一个cred结构体应用到进程。

那么内核题的流程就是：低权限程序 -> 与内核模块交互，进入内核空间让内核模块执行commit_creds(prepare_kernel_cred(NULL))，使当前进程变成root权限 -> 返回到用户空间，执行system('/bin/sh')新起一个shell（root权限）

跟内核模块进行交互

现在的问题就来到了怎么跟驱动程序进行交互，远程服务器会把驱动程序挂载到靶机上（对应的我们本地使用执行启动脚本时其也会将驱动程序挂载到qemu里）

Linux定义了系统调用ioctl供进程与设备之间进行通信，也就是说我们可以在程序中调用ioctl来跟某个驱动程序交互

系统调用ioctl是一个专用于设备输入输出操作的一个系统调用，其调用方式如下：

  • fd：设备的文件描述符
  • request：请求码
  • 其他参数

对于一个提供了ioctl通信方式的设备而言，我们可以通过其文件描述符、使用不同的请求码及其他请求参数通过ioctl系统调用完成不同的对设备的I/O操作

例如CD-ROM驱动程序弹出光驱的这一操作就对应着对“光驱设备”这一文件通过ioctl传递特定的请求码与请求参数完成

返回用户态的方法

使用swapgs;iretq指令返回用户态

swapgs指令简单地说就是将当前gs基址寄存器的值与一个特定地址（IA32_KERNEL_GS_BASE MSR）的值交换，从而快速获得指向内核空间/用户空间数据的指针，举个例子，程序运行在用户态时gs是用户空间的地址，此时IA32_KERNEL_GS_BASE MSR中保存的就是内核空间的地址，要进入内核态时执行swapgs，gs的值就变成了内核空间的地址，再需要返回到用户空间时同理。

iretq指令用来完成中断返回，其是中断服务程序中的最后一条指令，功能是将栈上的段地址、偏移地址pop到对应寄存器中，使程序返回原来发生中断的地方，其从栈顶弹出 EIP >> CS >> EFLAGS >> ESP >> SS （类似于RETF）

为什么需要手动返回用户态

正常情况下在程序中用利用ioctl函数去跟驱动进行交互时，函数结束完后当然会自动返回到用户态继续执行我们的程序，但是我们如果在内核态中利用栈溢出等漏洞劫持了PC指针之后，因为内核已经不走正常返回用户态的指令了，所以除了让其执行提权操作以外，我们还需要手动返回到用户态去开shell。

总结

照着Water Paddler队的wp复现的，不得不说外国师傅写wp都挺详细的，都落在重点上。这题我自己大概复现了两天左右，感觉是道很不错的题目

首先总结一下从中学到的知识点：

  1. 这题中用来泄漏libc的方式有点侧信道攻击的feel，利用运算结果的不同分别设置了两种跳转，对应着程序不同的输出结果，这样就可以通过程序的输出来判断当前泄漏的bit是多少

  2. 这题的漏洞为一个整数溢出导致的size跟实际malloc的size不同，从而获得了任意libc写，所以平时做题时可以重点关注有没有malloc的size跟输入的size有出入的情况，可能洞就在这里。

  3. get了一条在2.35下可用的任意函数执行的新链（___call_tls_dtors）

    利用条件总结一下就是：① 有libc(mmap)地址，有libc(mmap)任意地址写。② 能显式或者隐式调用exit。

    这条链的优点大概就是很容易构造，只需要修改很少的地方就可以做到任意函数执行且控制rdi。

    利用方式：①在fs:[-0x58]这个位置布置一个指针。②将fs:[0x30]这个点盖成0。③将要调用的函数地址rol 0x11之后写到①指针指向的地址。④在①指针指向的地址+8处布置参数地址。⑤调用exit

简述VMpwn

VM指的其实就是虚拟机，出题人会在题目中设计一部分代码用来模拟一台PC的运行，那么既然要模拟一台PC的运行，就免不了要涉及几个部分：①分配空间模拟代码段。② 分配空间模拟栈段。③分配空间模拟寄存器。④分配空间模拟数据段

这类VMpwn会接受一段我们输入的数据，并且把我们输入的数据按照它的规则解析成用来运行其伪汇编指令的代码，这种我们输入的数据被称为opcode（运行代码）

以一道题目举个例子

漏洞点

一、off-by-one

这题的重点在设置Username的函数处，其首先将输入直接写到临时的chunk中

然后通过这种方式来将临时chunk中符合条件的输入写到栈上的数组中，因为v10是用strcpy来写入的，所以v10数组中实际上会包含\x00（或者说v10数组的大小本身就决定了其会包含\x00），也就是说符合条件的输入一共有数字+大小写字符+\x00，这些符合条件的输入会被写到栈上同时会使v5计数增加

重点在最后，其用strdup将s数组中的内容写到了传进的User_content指针中，将计数v5直接写进了传进的User_len中

而strdup函数的定义如下，其是用strlen来确定对象的所需拷贝字节的

那么这时候应该也注意到问题了，我们输入的\x00会使计数 v5增加，但是在strdup内部其会将str截断使得malloc的大小与v5计数不符，这就会造成在后续修改一个字节用户名称的函数中造成一定范围内的off-by-one

二、地址泄漏

因为在SetUser和SetPasswd中都会申请临时chunk然后再free掉，这就导致在SetUser中被free掉的chunk可以在SetPasswd中被strdup malloc出来，使得passwd中可能有残留的指针，而且Passwd_len是用strlen来计算的，所以会导致残留的指针也被算进里面，在show时就可以被泄漏出来。

思路

用off-by-one可以直接修改chunk的size来造成over lapping（小改大），以前做题时遇到的单字节溢出一般都是off-by-null，所以习惯性地看到单字节溢出就会朝着unlink的那个方向去向，确实如果只有一个off-by-null的话要构造堆重叠往往就要配合unlink来实现向低地址合并，但是如果是off-by-one的话想要构造堆重叠就大可不必用到unlink了。

然后就是堆风水的问题了，堆风水的细节其实不好描述，根据malloc的流程去仔细想想就行，这题其实还是比较值得以后再来复现学习一下的。

我们都知道延迟绑定技术依赖于_dl_runtime_resolve函数（64位下是_dl_runtime_resolve_xsavec），如果只概括地说，那么这个函数所完成的功能就是：

  1. 解析函数的真实地址
  2. 将解析所得的真实地址写到函数对应的got表里面

而如果要说得详细一些，则可以把_dl_runtime_resolve_xsavec的调用链拿出来：

首先_dl_runtime_resolve_xsavec定义在dl-trampoline.h里，是拿汇编写的，其功能可以概括为：

  1. 在栈上保存大部分寄存器的值
  2. 将之前push进来的link_map还有reloc_arg赋值给rdi跟rsi，然后调用_dl_fixup

思路

看到这题的主体时很自然能够想到是要借助申请的堆来做到任意libc地址写，那么首先第一个问题就是申请到mmap段上的堆至少需要malloc一个0x21000左右的堆，也就是申请的size肯定是大于两个字节的，那么正常来说执行完一次之后就会跳出循环。为了避免这种情况就需要利用延迟绑定，来让解析完的write的真实地址去覆盖掉size的高位，进而得到无限次数的libc地址写。

那么第一步已经在前面说过方法了，可以通过修改link_map的第一个成员变量l_addr来完成，这题size的地址是0x408C，write的got表是0x4018，那么只需要将l_addr的低位改成0x408C - 0x4018 - 4 = 0x70即可

计算tc_idx

大概梳理一下malloc的流程，在有tcache的版本下，程序首先会根据传入的参数大小（bytes）计算出实际需要的chunk大小（tbytes），然后根据这个大小计算出对应的tcache bin下标（tc_idx）

因为在攻击中我们需要知道这个tc_idx和tbytes的对应关系，所以要了解一下具体是怎么算出来的

其中MINSIZE一般在32位下是0x10字节，64位下是0x20字节，MALLOC_ALIGNMENT在32位下是8个字节，64位下是16个字节。

那么这样代入我们就能算出特定大小的chunk所对应的tc_idx了

相关结构

然后程序根据以下三点来判断是否从tcache中取块：① 下标在tcache bin范围里。② tcache指针有值。③ 下标对应的bin中有chunk

然后tcache_get函数是这样的

这个函数很简单，根据下标从tcache->entries中取chunk地址给e，更新tcache->counts[tc_idx]，然后tcache->counts[tc_idx]减一。

这里的tcache是个结构体指针，其指向对应线程所维护的那个tcache_perthread_struct结构体

其中tcache_perthread_struct是整个tcache的管理结构，其中记录了每条tcache bin链表中chunk的数量以及每条tcache bin链表头部的地址，这个结构体的总大小为1 * 64 + 8 * 64 = 0x240个字节，而每个 thread 都会用一个chunk来存放这个管理结构，所以其总大小为0x250

对于主线程来说，其位于heap段开头。

值得注意的是，从glibc-2.30开始，tcache_perthread_struct中的counts变成了uint16_t类型的，也就是说一个counts占两个字节，tcache_perthread_structchunk的总大小变成0x290

利用

在了解完上面的点之后其实可以发现，由于tc_idx是直接跟tbytes挂钩的，所以当tbytes大于正常的tcache bin chunk时，tc_idx就不会是正常的tcache bin下标，但是由于其存在对下标合法性的检查（如下），所以正常情况下当tc_idx不在tcache bin范围中时，程序就不会从tcache中取chunk。其实也可以理解为“程序是通过计算出来的下标是否属于tcache bin以及tcache->counts[tc_idx]是否大于0”来决定“是否从tcache bin中取chunk”。

所以如果我们修改mp_.tcache_bins为一个大值，那么程序就会认为一个大范围内的地址都属于tcache bin，而且由于主线程的tcache_perthread_struct结构就在heap段开头，所以tcache_perthread_struct之后的地址中的值都是可控的，这也就意味着我们可以直接往伪造的tcache bin上放任意地址，然后申请对应的大小，从这个伪造的tcache bin中将这个我们布置的地址给拿出来。只需要同时保证对应的tcache->counts[tc_idx] > 0即可。
