动态链接相关

静态链接程序和动态链接程序

静态链接的文件将所有所需的库函数都写入了文件内，动态链接的文件只是将所需要用的库函数标记，在需要用时再在外部动态链接文件中寻找，动态链接的文件本身储存程序的主逻辑即可，这样做能减少大量文件所需的空间

动态链接库由装载器将其载入内存中，装载器首先会将自己载入内存，然后它就能将其他动态链接库载入内存中

静态链接的文件往往有很多系统调用碎片可以组成一次systemcall

在elf文件的.code段，有一个plt节，这个节里保存了一些用来解析存放在libc code段里的某个函数的真实地址的代码，解析完毕后将其填入.data段里的got.plt节，然后以后再次调用这个函数时就直接从got.plt节里获取这个函数的真实地址，然后跳转到这个地址中执行函数

动态链接的程序有一些特有的结构：

所有动态链接库的可执行文件会在link_map中形成一个链表保存起来

__dl_runtime_resolve就是用来解析某个函数在库函数中真实地址的函数

.got（全局偏移量表）是代码段中的一个节，其保存了整个程序的虚拟内存空间中的全局变量的偏移

.got.plt节保存函数偏移，其在数据段

ret2syscall

ret2syscall即利用程序中的gadget控制寄存器参数后执行系统调用，从而获得shell

知识点

  • 32位平台系统调用使用”int 80”，64位平台系统调用使用”syscall”

    同时32位和64位平台的系统调用函数的调用号不一样，使用的时候最好先查一下

  • 32位的系统调用号放在eax 传参依次是 EBX、ECX、EDX、ESI、EDI、EBP

    64位的系统调用号放在rax 传参依次是 RDI、RSI、RDX、R10、R8、R9 （和64位函数传参一样）

system(“/bin/sh”)实际上是execve(“/bin/sh”,0,0)的一个集成，实际上前者最终也是调用了后者获得的shell

在进行系统调用之前需要为用到的寄存器赋值，在进行系统调用后才能正常调用函数，这里就需要通过程序中的gadget来控制寄存器参数

思路

  • 在栈溢出的基础上通过一系列的pop|ret指令来构成ROP链为目标寄存器赋上对应的值，然后执行系统调用即可

ret2libc

当题目中没有出现system函数和’/bin/sh’字符串时，就需要通过泄漏libc地址来寻找libc中的system函数和’/bin/sh’字符串来以此执行system(“/bin/sh”)

思路：在栈溢出的基础上控制程序返回到一个输出函数中，并将某个函数got表的地址作为其参数传递，这样就可以通过输出函数来泄漏某个函数的真实地址（存在于libc中），要注意的是由于延迟绑定机制，要泄漏的函数必须是已经执行过的函数。泄漏完成后控制程序返回再次触发栈溢出，这时由于已经获得了system函数的地址及”/bin/sh”，就可以获得shell

综上，步骤即：

  • 触发栈溢出，控制程序执行输出函数（返回到输出函数在plt表的地址）
  • 将已经执行过的函数作为输出函数的参数传递，以此打印出该函数的真实地址
  • 函数的真实地址 - 函数在libc中的偏移 = libc的基地址
  • libc的基地址 + 函数libc中的偏移 = 函数真实地址
  • 控制程序返回，再次触发栈溢出，执行system(“/bin/sh”)
