知识点

原理在 ctf-wiki 上讲得很清楚了，这里不再重复

首先需要存在已知如何触发的栈溢出，而且需要远程的进程服务崩溃后不会re-rand

然后需要爆破出溢出所需的padding大小

若存在Canary则需要按字节爆破，例如Canary为 0x7766554433221100 ，则从倒数第二个字节开始逐字节爆破，即 padding + 0x0100 ，若Canary报错则说明倒数第二个字节错误，爆破至正确后继续爆破倒数第三个字节直至得到Canary值

我们需要根据程序的行为来猜测我们寻找的地址应该是什么，为此我们需要先找到 stop gadget

stop gadget

当我们把返回地址覆盖成我们随意选取的内存地址时，程序会有很大的概率crash，但是也存在一种情况，即当我们控制程序返回到一块内存区域后，程序并没有crash，而是hang在那里，使我们能一直保持连线状态，我们将这种gadget称为 stop gadget 。同样我们需要通过爆破来寻找这样的 stop gadget

Brop gadget

我们需要将远程服务器dump出程序的内存到本地，为此我们需要一个输出函数，还要能控制对应寄存器的gadget，由于即便找到了 pop;ret 这样的gadget也很难知道其pop的是哪个寄存器，所以我们更倾向于找到__libc_csu_init 中的 pop6_ret（这里称为Brop gadget） 的地址，这样即可准确控制寄存器的值，另外 Brop gadget 往下偏移9即可得到 pop rdi;ret ，同理 Brop gadget 往下偏移7得到 pop rsi;pop r15;ret;

寻找plt表

为找到输出函数，我们需要找到plt表，plt表具有比较规整的结构，每一个plt表项都是 16 字节，且每个表项的6字节偏移处，是对该表项对应的函数的解析路径，即程序最初执行该函数的时候，会执行该路径对函数的 got 进行解析，如下

所以 return 到 plt 表项的开头时程序就会调用该函数（无论是否解析过），而 return 到 plt 表项 +6 处，则会再进行一次函数的解析，这也无所谓，因为最终同样会调用该函数。

当我们发现好多条连续16个字节对齐的地址都不会造成crash，而且这些地址+6的地址同样不会造成crash，那么我们有理由相信这就是某个plt对应的表项

错位构造gadget

发表于 2022-01-30

pop rdi;ret;构造

在 __libc_csu_init 函数内的 pop r15;ret; ，它的字节码是41 5f c3。

而pop rdi;ret;的字节码是5f c3。

那么当EIP指向 pop r15;ret; 的地址 +1 时，程序事实上将会执行pop rdi;ret;。

pop rsi;pop r15;ret;构造

同理在 __libc_csu_init 函数内的pop r14;pop r15;ret;，它的字节码是41 5e 41 5f c3。

而pop rsi;pop r15;ret;的字节码是5e 41 5f c3。

那么当EIP指向 pop r14;pop r15;ret; 的地址 +1 时，程序事实上将会执行pop rsi;pop r15;ret;。

程序中有时会有一些可利用的 gadget 直接在IDA中无法直接找到，因为同样是用错位得到的，这时可以用ROPgadget等工具尝试寻找。

正常情况下main()函数被调用后main()函数随即创建其栈帧，所以返回地址一般在ebp + 4处，但是我们观察main()函数开头的汇编指令可知main()函数在创建其栈帧前还多了几步操作，这使得返回地址离ebp的距离有所不同

可以看到esp的最低4位是c，所以最后ebp相对于返回地址的距离为0xc + 0x4 + 0x4 = 0x14个字节，故数组v13相对与返回地址的距离为0x14 + 0x70 = 0x84个字节。

这是一道拟态的题目，给了两个二进制文件，分别是32位和64位的，两个程序功能完全相同。题目会同时启动32位和64位程序，然后将我们的输入分别传给这两个进程，并检测两个程序的输出，若两个程序输出不同或者其中一个程序崩溃则会判断为check down，直接断开连接，只有两个程序的输出一致且都不崩溃时，才能通过检查。所以我们需要构造一条在两个程序中都能getshell的payload。

分析

显然两个程序的漏洞都为栈溢出，且在64位程序中 buf 到 rbp 的距离为 0x110 ，而在32为程序中 buf 到 ebp的距离为 0x10C ，所以对于同一条payload而言，64位程序的padding就为 0x118 个字节，而32位程序的padding为 0x110 个字节。因为这两个程序都是静态链接的程序，所以我们基本可以找到所需要的gadget来执行syscall，我们利用 add_sp_ret 这样的gadget来分别抬高两个程序中的sp寄存器，这样就能在栈上分别布置32位和64位的ROPchain，且使两个程序只执行对应的ROPchain。

因为32位程序会先ret（buf到ret的距离较短），所以我们原本可以利用32位程序中的add_sp_ret来抬栈从而使32位程序的执行流直接跳过64位的ROPchain，但是32位程序中没有能将 esp 直接抬高这么多的 gadget，所以这种思路不可行，但是这不重要，只要我们能将 esp 抬高到跳过64位程序的 ret 就行，然后我们使64位程序的 ret 同样布置为 add_sp_ret 这样的gadget，来使这条payload在64位程序中的执行流跳过32位程序的ROPchain即可。同时因为程序中没有 /bin/sh 字符串，所以我们还需要往程序里手动写入。结合exp来理解即可，exp如下：

NX保护

作用

将数据（堆、栈）所在内存页标识为不可执行，当程序尝试在数据页面上执行指令时，CPU会抛出异常而不是去执行指令。

编译选项

关闭：-z execstack

开启：-z noexecstack

PIE保护

作用

使程序每次加载的基址随机化

编译选项

关闭：-no-pie

开启：-pie-fPIC

Canary保护

作用

函数开始执行时往栈上插入Canary值，并在函数返回的时候检查Canary的值是否有变动，如果有则调用 __stack_check_fail 函数强行停止程序。

编译选项

关闭：-fno-stack-protector

部分启用（只为局部变量中含有 char 的函数插入Canary）：-fstack-protector

全局启用（为所有函数插入Canary）：-fstack-protector-all

Fortify保护

作用

主要用来防止格式化字符串漏洞。包含 %n 的格式化字符串不能位于程序内存中的可写地址。当使用位置参数时，必须使用范围内的所有参数（例如要使用 %7$x ，则必须同时使用 1$，2$，3$……）

编译选项

关闭：-D_FORTIFY_SOURCE = 0

开启：-D_FORTIFY_SOURCE = 2

RELRO保护

作用

设置符号重定向表为只读并在程序启动时就解析并绑定所有动态符号，从而减少对GOT表的攻击

编译选项

开启（部分）：-z lazy

开启（完全）：-z now
