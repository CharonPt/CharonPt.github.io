retfq指令

程序通过cs寄存器来确定以64/32位模式来运行，cs = 0x23对应32位模式，cs = 0x33对应64位模式，而retfq则是用来修改cs寄存器的，其功能相当于ret;pop cs。

绕过沙箱

当沙箱的第一行是A = sys_number时，我们可以通过切换程序运行的位数来调用到在另一个位数中相同调用号的函数。如下图中，程序正常时工作在64位模式下，则这个沙箱过滤的就是64位下这些函数的系统调用号，也就是只允许调用5（fstat）、37（alarm）、0（read）、9（mmap）、231（exit_group）这几个系统调用，但是这里沙箱是只允许调用这些调用号，而不是只允许调用这些函数，同时我们知道32位和64位下的系统调用号对应的是不同的函数，例如64位中5号系统调用对应的是fstat函数，但是32位中5号系统调用对应的是open函数。所以我们有时就能通过切换程序工作位数的方法来调用到一些沙箱原本过滤掉的函数。

侧信道攻击的概念

侧信道信息一般分成：声音、温度、功耗、电磁、色彩、等，在加密硬件进行加密的时候，上述的信息只是加密过程中附带产生的物理量，这列物理量不会对加密提供任何好处，攻击利用收集到的这些物理信息，通过结合统计学手段，或者其他算法揭秘硬件上使用的秘钥。这些利用物理量达到破解秘钥的方法就叫做侧信道攻击。

我的理解是侧信道攻击就是通过对一些加密信息本身以外的信息来进行分析，从而得到我们想要的加密信息本身的攻击方式，这一块算是一些别的知识拓展，对做题本身而言没有帮助

SIGFPE信号

“错误的算术操作”会使得SIGFPE信号被发送给进程，例如：整数除零、整数上溢、浮点除零、浮点上/下溢……

一个通常的疏忽是认为除以零是SIGFPE的唯一来源。在一些架构上（包括IA-32[来源请求]），使用INT_MIN（最小的可以被表示的负整数值）除以-1的整数除法也会触发这个信号，因为商是一个无法被表示的正数。（比如8位有符号整数可以表示-128、+127和它们之间的整数。-128÷-1=+128 > +127，因此无法被表示而产生溢出并触发此信号）

符号执行-基于python的二进制分析框架angr

发表于 2022-02-13 更新于 2022-02-16

符号执行

[一篇比较好的文章](symexe/klee_analysis.md at master · chyyuu/symexe · GitHub)

如果仅学习如何使用angr框架的话其实不需要对符号执行有过多了解，所以这里只做简单概述，更详细的地方可以看上面的文章。

符号执行是指在不执行实际程序的前提下，把源程序翻译为一种中间语言，用符号值表示程序变量的值，然后基于中间语言模拟程序执行来进行相关分析的技术，它可以分析代码的所有语义信息，也可以只分析部分语义信息（如只分析“内存是否释放”这一部分的语义信息）。

程序的路径（path）是程序的一个语句序列，这个语句序列包括程序的一些顺序的代码片段，代码片段之间的连接是由于分支语句导致的控制转移。一个路径是可行的（feasible），是指存在程序输入变量的至少一组值，如果以这组值作为输入，程序将沿着这条路径执行。否则，路径就是不可行的（infeasible）。路径条件（path condition,PC）是针对一个路径的，它是一个关于程序输入变量的符号值的约束，一组输入值使得程序沿着这条路径执行当且仅当这组输入值满足这条路径的路径条件。

例如某个程序在执行过程中会执行如下语句

angr的作用就在于能够得出上述的使得程序走”可行路径“的”至少一组值“。

要注意的是符号执行的路径选择问题到现在依旧是一个很大的问题，换句话说也就是当我们的程序存在循环时，因为符号执行会尽量遍历所有的路径，所以每次循环之后会形成至少两个分支，当循环的次数足够多时，就会造成路径爆炸，整个机器的内存会被耗尽。

前言

ret2_dl_resolve

大概调用过程：

触发了_dl_runtime_resolve之后会

  1. 先到.rel.plt节，通过plt中给的push <偏移>在.rel.plt节中找到偏移处对应函数的重定位表项
  2. 再通过该表项的r_info（一个偏移量），找到.dynsym节中对应偏移量函数的符号表项
  3. 又通过该符号表项的st_name（也是一个偏移量）在.dynstr节找到对应的字符串
  4. 最后就会通过这个字符串（函数名）来调用该函数。

以下示例中的程序为32位，64位的流程未必一样

  1. call foo -》 jmp foo@plt

  2. foo@plt的内容一定是 jmp [foo@got]，意为取foo@got表项中的内容来跳转

  3. 在函数未进行过延迟绑定的解析时，foo@got表项中的内容会被填上foo@plt+6的地址，如下，其中0x804b22c是read@got的位置，此时里面的内容正是read@plt+6

    在IDA中观察一下plt表，可以发现所有函数的plt表项都是类似的操作：

    • X@plt+0处是jmp [X@got]
    • X@plt+6处是push index（这个index是该函数在rel.plt上的偏移.reloc_arg，_dl_runtime_resolve的第二个参数）
    • X@plt+B处是jmp plt0（plt0指的是plt表的第一项，是_dl_rumtime_resolve的表项）

  4. plt0中的内容是固定的

    • push [GOT + 4]（link_map结构的地址，_dl_runtime_resolve的第一个参数）
      • link map这个结构我们不关心，但是要知道它包含了.dynamic的指针，通过这个link_map，_dl_runtime_resolve函数可以访问到.dynamic这个section
    • jmp [GOT + 8]（_dl_runtime_resolve的地址），此时相当于开始执行_dl_runtime_resolve(link_map,reloc_arg)）

综上，我们可以总结出两种思路

方法一（需要知道libc版本）

如果我们要绕过第一个if，即 if (__builtin_expect (ELFW(ST_VISIBILITY) (sym->st_other), 0) == 0)

那么我们最后会走的路径就是value = l->l_addr + sym->st_value，那么如何让value变成我们想要执行的函数呢？

我们需要

  1. 让l->l_addr设置成目标函数跟一个程序中已解析函数真实地址的偏移
  2. 让&(sym->st_value)为一个程序中已解析函数的got表的地址

但是，显然这种方法必须要知道libc版本，因为我们需要使用一个提前已知的函数偏移

现在假设我们期望执行system函数

例如程序中存在setbuf函数，且能在我们的攻击流程前完成解析，那么我们就需要让l->l_addr = system - setbuf

然后让sym + st_value = setbuf@got，即sym = setbuf@got - 8。这里相当于我们把sym放在了got表的附近，利用里面天然的数据来完成利用，而且正常情况下任何一个函数的got表项-8的位置都不可能是0，因为got0处会存link_map和dl_runtime_resolve的地址，也就天然绕过了第一个if。

基于这样的设置，value = offset + setbuf_addr = system_addr，我们就完成了利用

可以想到这样的方法相对来说方便一点，我们甚至不需要伪造sym结构，只要计算好reloc_arg即可。

但是也能想到既然我们需要伪造l->l_addr，那么我们逃不开要伪造link_map中必要的字段例如l->l_info[]，好在整个流程中用到的link_map的字段并不算多，我们只要伪造很小的一部分即可

知识点

原理在 ctf-wiki 上讲得很清楚了，这里不再重复

首先需要存在已知如何触发的栈溢出，而且需要远程的进程服务崩溃后不会re-rand

然后需要爆破出溢出所需的padding大小

若存在Canary则需要按字节爆破，例如Canary为 0x7766554433221100 ，则从倒数第二个字节开始逐字节爆破，即 padding + 0x0100 ，若Canary报错则说明倒数第二个字节错误，爆破至正确后继续爆破倒数第三个字节直至得到Canary值

我们需要根据程序的行为来猜测我们寻找的地址应该是什么，为此我们需要先找到 stop gadget

stop gadget

当我们把返回地址覆盖成我们随意选取的内存地址时，程序会有很大的概率crash，但是也存在一种情况，即当我们控制程序返回到一块内存区域后，程序并没有crash，而是hang在那里，使我们能一直保持连线状态，我们将这种gadget称为 stop gadget 。同样我们需要通过爆破来寻找这样的 stop gadget

Brop gadget

我们需要将远程服务器dump出程序的内存到本地，为此我们需要一个输出函数，还要能控制对应寄存器的gadget，由于即便找到了 pop;ret 这样的gadget也很难知道其pop的是哪个寄存器，所以我们更倾向于找到__libc_csu_init 中的 pop6_ret（这里称为Brop gadget） 的地址，这样即可准确控制寄存器的值，另外 Brop gadget 往下偏移9即可得到 pop rdi;ret ，同理 Brop gadget 往下偏移7得到 pop rsi;pop r15;ret;

所以 return 到 plt 表项的开头时程序就会调用该函数（无论是否解析过），而 return 到 plt 表项 +6 处，则会再进行一次函数的解析，这也无所谓，因为最终同样会调用该函数。

当我们发现好多条连续16个字节对齐的地址都不会造成crash，而且这些地址+6的地址同样不会造成crash，那么我们有理由相信这就是某个plt对应的表项
